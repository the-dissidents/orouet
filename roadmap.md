# Architecture Design Document: AI-Assisted Literary Translation Tool

## 1. System Overview

A local-first, AI-assisted desktop application designed for literary translation. The system provides granular structural alignment, branching version control, and contextual AI assistance without relying on rigid text segmentation or polluting the text with visible markup.

* **Framework:** Tauri (Local Desktop Application)
* **Frontend Core:** Svelte, TypeScript, ProseMirror
* **Backend Core:** Rust
* **Local Database:** Embedded SQLite (`rusqlite` + `sea-query`)

---

## 2. Data Architecture: Event-Sourced DAG

### The Abstract Syntax Tree (AST)

* **Format:** The application abandons raw strings and Markdown. All text is represented as ProseMirror JSON objects.
* **Advantage:** Prevents CJK formatting artifacts, allows precise index mapping, and natively generates `Steps` (JSON diffs) for every user action.

### Version Control (The DAG)

* **Structure:** A unified Directed Acyclic Graph (DAG). Every text block (both Source and Target) exists within the same global version control system.
* **Payload:** Nodes (`Commits`) do not store full text strings. They store `Deltas`—arrays of ProseMirror JSON `Steps`.
* **State Rehydration:** To view historical states, the Rust backend traverses the DAG via Breadth-First Search (BFS) and applies the JSON steps sequentially.
* **Session Batching:** To prevent graph pollution from individual keystrokes, the Svelte frontend debounces typing and sends batched arrays of ProseMirror steps to Rust only upon session triggers (blur, idle, or explicit branching).

---

## 3. Storage & Metadata

### Embedded SQLite

* **Mechanism:** Uses synchronous `rusqlite` wrapped in a Rust `Mutex` for safe concurrent access. `sea-query` is used for dynamic, type-safe SQL generation without the bloat of an async runtime or strict ORM.
* **Lazy Loading:** Prevents memory bloat by only fetching the specific DAG commits required for the user's current viewport.

### Sidecar Metadata (Annotations)

* **Separation of Concerns:** Editorial notes and reminders are **not** injected into the text AST.
* **Implementation:** They are stored in a dedicated SQLite table. They are linked to the text via a lightweight ProseMirror `mark` containing only a UUID.

## 4. Structural Layout & UI

### The "Cluster" Wrapper

* **Problem Solved:** 1:1 paragraph mapping fails in literary translation (e.g., splitting one long archaic paragraph into three modern ones).
* **Implementation:** A `Cluster` struct wraps arrays of Source Node IDs and Target Node IDs.
* **CSS Grid:** The UI uses CSS Grid where each `Cluster` is a row. This natively handles asymmetrical heights (1:N or N:1 mappings) without fragile JavaScript scroll-syncing.

### Dynamic Projection Views

* **Contextual Inspector (Column 3):** Dynamically scopes its content (LLM Chat context, Dictionary, or Annotation threads) exclusively to the active Cluster.
* **DAG Overlay:** Version control history is projected dynamically. Svelte queries Rust for a localized sub-graph affecting only the selected paragraph, rendering it in an interactive modal.

## 5. Machine Learning Alignment Pipeline

### Semantic Engine

* **Technology:** Local Cross-Lingual Encoder (e.g., `SimAlign` or `Awesome-align`) running via the ONNX runtime (`ort` crate) in Rust.
* **Mechanism:** Maps tokens to high-dimensional semantic coordinate vectors. Aligns words by calculating physical distance in vector space, agnostic to grammatical restructuring. Costs zero LLM tokens.

### UX & Overrides

* **Implicit Operation:** Alignment boundaries are maintained mathematically in the background. No visible `[Tags]` or IDs are exposed to the user in the standard editor.
* **Alignment Overlay Mode:** Because literature contains idioms that confuse standard encoders, users can toggle an interactive SVG mode to manually sever or connect word-chip coordinates.

## 6. Security & Key Management

### Encrypted Vault

* **Technology:** `tauri-plugin-stronghold` to bypass OS-specific keychain restrictions (like macOS Access Groups) and ensure uniform cross-platform behavior.
* **Zero-Prompt UX Strategy:** To avoid forcing users to enter a master password on every launch for an open-source app, the Stronghold vault encryption key is derived via **Salted Hardware Fingerprinting**.
* **Mechanism:** Rust combines a hardware identifier (e.g., MAC address) with a static application salt, hashes it via Argon2, and unlocks the local API key vault silently.
